# Story 2.8 — Unifier les écritures atomiques vers `fsutil.WriteFileAtomic`

**Sprint :** 🟠 2 — HIGH
**Priorité :** HIGH
**Effort :** 1h
**Agent assigné :** Forge (Dev)
**Dépendances :** Aucune
**Fichiers :** `multiai-go/internal/openrouter/cache.go`, `multiai-go/internal/config/wizard.go`

---

## Contexte

Le pattern temp-file + fsync + rename est implémenté 4 fois différemment dans le code. Deux utilisent `fsutil.WriteFileAtomic`, deux ont leur propre implémentation (dont une avec un nom de temp fixe = race condition).

Source : Audit architecture R2.

## Ce qu'il faut faire

### Étape 1 — Remplacer dans `openrouter/SaveCache()`

```go
// Avant (race condition : nom de temp fixe)
tmpPath := path + ".tmp"
os.WriteFile(tmpPath, data, 0644)
os.Rename(tmpPath, path)

// Après
fsutil.WriteFileAtomic(path, data, 0644)
```

### Étape 2 — Remplacer dans `config/setEnvVarInFile()`

Vérifier si `setEnvVarInFile` peut utiliser `WriteFileAtomic` directement, ou si la logique de parsing .env doit être conservée (dans ce cas, écrire dans un temp via `os.CreateTemp` puis utiliser `WriteFileAtomic`).

### Étape 3 — Vérifier la cohérence

```bash
cd multiai-go
rg -n 'ioutil\.WriteFile|os\.WriteFile.*Rename|os\.CreateTemp' --glob '*.go' --glob '!*_test.go'
```

Toute occurrence doit :
- Soit utiliser `fsutil.WriteFileAtomic`
- Soit avoir une bonne raison documentée de ne pas le faire

### Étape 4 — Tester

```go
func TestWriteFileAtomic_AllCallers(t *testing.T) {
    // Vérifier que config, secret, openrouter utilisent tous WriteFileAtomic
}
```
